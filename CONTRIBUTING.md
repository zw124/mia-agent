# Contributing to Mia

Thanks for your interest in contributing.

## Before you contribute

- Read [CLA.md](./CLA.md).
- Read [CODING_STANDARDS.md](./CODING_STANDARDS.md).
- Read [DEVELOPMENT.md](./DEVELOPMENT.md) and [TESTING.md](./TESTING.md).
- Check open issues and existing design discussions before starting large work.

## Contribution model

Mia uses:

- `AGPL-3.0-only` for the community edition;
- a separate commercial licensing path for proprietary use;
- a contributor license agreement so the project can be maintained under both tracks.

## How to contribute

1. Fork the repository or create a topic branch.
2. Keep changes scoped to one problem.
3. Add or update tests for behavior changes.
4. Update docs when user-facing behavior changes.
5. Open a pull request using the PR template.

## Pull request expectations

Every pull request should:

- explain the problem and the chosen solution;
- describe risks, migrations, or compatibility impact;
- include screenshots for UI changes;
- pass CI and required status checks;
- avoid unrelated formatting churn.

## What maintainers may reject

Maintainers may reject contributions that:

- add legal or licensing conflicts;
- reduce security posture;
- introduce unreviewed dependencies;
- bypass approval or auditing workflows;
- conflict with the product boundary between community and commercial offerings.

## Communication

- Use GitHub Issues for bugs and feature requests.
- Use Discussions or issues for architecture proposals if enabled.
- Use security channels only for confidential vulnerability reports.
